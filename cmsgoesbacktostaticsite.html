Classic Content Management Systems Will Die

Remember when websites were built using static html documents? Looks like these times are back!

In the late 1990s, CMS' became increasingly popular and their success story really took off in the 2000s. With a CMS, websites are built using templates, databases, plugins and application code to dynamically assemble the web pages when requested by users. Because various applications need to work together, this created a complex system, with several applications running at the same time.

The bigger and more complex websites became, the more difficult it got to keep up with the complexity. Distributing content over various databases, caching application output and handling traffic peaks has to be scaled. All these server applications add a lot of clutter and make your website vulnerable to outings and hackings. A typical stack consists of about 20 or more software components and the attack surface increases significantly by adding more and more gaddgets or apps, as people tend to do.

So: do we really need dynamic websites? Or isn't most content we publish static? In our opinion, more than half of the websites that are currently published with CMS’ could very well do with static websites, while additionally saving money and time and achieving higher security standards. Static Site also offer more scalability. Round-robin DNS can be easily introduced to use as many webservers as needed. There are no complex session caching and frontend node binding rules across multiple frontend servers, etc.
Of course, expanding your website will be dependent on development of external services and their API’s (like Imgur for images, Disqus for commenting functions, Vimeo or Wistia for videos, etc.). In the long run, though, it’s perfectly possible to build any server-side application and fully integrate it into your static website. Especially on an open source basis, alle these services are really bound to take off.
Another phenomenal reason for static sites is the migration of your content. No more complex migration scripts. Just set up a Git repo and transfer all your content to another solution whenever you like!
